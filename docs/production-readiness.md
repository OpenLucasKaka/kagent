# Production Readiness

This checklist captures the current release posture for the self-correcting
LangGraph agent service.

## Release Gates

Run these before publishing or deploying:

```sh
scripts/run_checks.sh
scripts/smoke_service.sh
scripts/smoke_internal_runtime.sh
scripts/staging_acceptance.sh \
  >/tmp/self-correcting-agent-staging-acceptance.json
scripts/observability_acceptance.sh \
  >/tmp/self-correcting-agent-observability-acceptance.json
scripts/internal_rollout_acceptance.py \
  --signoff /tmp/self-correcting-agent-internal-rollout-signoff.json \
  >/tmp/self-correcting-agent-internal-rollout.json
self-correcting-agent-doctor --trace-dir /tmp/self-correcting-agent-traces
self-correcting-agent-doctor --require-auth --trace-dir /tmp/self-correcting-agent-traces
self-correcting-agent-doctor --production --trace-dir /tmp/self-correcting-agent-traces
SELF_CORRECTING_LLM_BASE_URL="${PROVIDER_BASE_URL}" \
SELF_CORRECTING_LLM_API_KEY="${PROVIDER_API_KEY}" \
SELF_CORRECTING_LLM_MODEL="agent-runtime-model" \
SELF_CORRECTING_SERVICE_RUNTIME_MAX_ITERATIONS=2 \
self-correcting-agent-doctor --production --require-runtime-provider \
  --trace-dir /tmp/self-correcting-agent-traces
scripts/smoke_real_llm_runtime.sh \
  >/tmp/self-correcting-agent-provider-smoke.json
scripts/production_readiness_audit.py \
  --provider-smoke-evidence /tmp/self-correcting-agent-provider-smoke.json \
  --require-provider-smoke \
  --staging-acceptance-evidence /tmp/self-correcting-agent-staging-acceptance.json \
  --require-staging-acceptance \
  --observability-acceptance-evidence /tmp/self-correcting-agent-observability-acceptance.json \
  --require-observability-acceptance \
  --internal-rollout-evidence /tmp/self-correcting-agent-internal-rollout.json \
  --require-internal-rollout
self-correcting-agent-release-evidence \
  --run-checks-exit-code 0 \
  --readiness-audit /tmp/self-correcting-agent-production-readiness-audit.json \
  --release-manifest /tmp/self-correcting-agent-release-manifest.json \
  --provider-smoke-evidence /tmp/self-correcting-agent-provider-smoke.json \
  --require-provider-smoke \
  --staging-acceptance-evidence /tmp/self-correcting-agent-staging-acceptance.json \
  --require-staging-acceptance \
  --observability-acceptance-evidence /tmp/self-correcting-agent-observability-acceptance.json \
  --require-observability-acceptance \
  --internal-rollout-evidence /tmp/self-correcting-agent-internal-rollout.json \
  --require-internal-rollout \
  --output /tmp/self-correcting-agent-release-evidence.json
scripts/production_approval_bundle.sh --strict
```

The standard gate covers tests, Ruff, byte compilation, CLI smoke checks, real
HTTP service smoke checks, internal subject runtime smoke checks, deployment
doctor self-checks with runtime policy fingerprints, evaluator checks, metrics checks, no-build-isolation wheel
build, isolated PEP 517 wheel build,
release manifest generation with wheel `sha256` hashes, manifest `verify`
checks, `package mismatch` and `version mismatch` detection,
`artifact_count mismatch` detection, `artifacts must be a list` and
`artifact entry must be an object` schema detection, `artifact path missing`
and `artifact path invalid` path validation, `artifact is not a file`
detection, and a clean virtual-environment install smoke for the built wheel.
The release evidence bundle generated by
`self-correcting-agent-release-evidence` is the handoff artifact for internal
approval. It records the `scripts/run_checks.sh` exit status, production
readiness audit status, release manifest verification result, optional provider
smoke status and run IDs, evidence file hashes, package version, and UTC
generation time without copying provider secrets into the repository or the
bundle. External evidence files are also scanned before the bundle can become
ready; secret-like evidence fields block release with `evidence_secret_detected`
and only redacted `evidence_secret_findings` containing label, JSON path, and
reason are written to the bundle.
`scripts/production_readiness_audit.py` applies the same external evidence
secret scan, so provider, staging, observability, and rollout evidence are
blocked before the final release bundle is generated.
`scripts/production_readiness_audit.py` also emits `openapi_contract` evidence
with the OpenAPI contract hash, path count, operationId count, duplicate
operationId detection, and required runtime path coverage. Archive that JSON
with release evidence so internal client integrations can tie generated SDKs
and change reviews back to one API contract.
The audit emits `configuration` evidence with an env example semantic check.
It records `service_keys_present`, `runtime_keys_present`,
`provider_keys_present`, missing key lists, key count, and template `sha256`,
so release evidence proves the packaged environment template still covers
service controls, Codex-style runtime controls, and OpenAI-compatible provider
configuration without storing provider secrets.
The audit emits `integration` evidence with an internal runtime client semantic check.
It records `commands_present`, `auth_present`, `idempotency_present`,
`runtime_routes_present`, audit-field coverage, secret-marker checks, and client
file `sha256`, so release evidence proves the packaged Python client example
still demonstrates Bearer auth, idempotent `/runtime/run`, `/runtime/resume`,
`/runtime/policy`, `/runtime/runs`, `/runtime/runs/summary`, approved action
IDs, auth subject filters, fleet summary filters, runtime policy audit, and
resume audit fields for internal teams. The client check also requires
effective policy filtering markers such as `effective_tool_policy`,
`--tool`, and `--approval-required`, so internal operators can inspect a
single tool's current approval boundary without custom code.
The same audit emits `observability` evidence with a Grafana dashboard semantic
check. The Grafana dashboard semantic check records `required_metrics_present`,
panel count, dashboard title, missing metric names, and dashboard `sha256`, so
production promotion fails if the packaged dashboard no longer covers the core
service, runtime latency, approval, stale pending approval gauges,
per-subject usage, per-subject outcome, resume, and tool-error metrics,
including `self_correcting_agent_runtime_stale_pending_approvals_current`.
It also runs a Prometheus alert rules semantic check that records
`required_alerts_present`, `required_metrics_present`, alert count, missing
alert names, missing metric names, and rules file `sha256`. This keeps service
down, HTTP error, runtime latency, stale pending approval, per-subject
failure/resume, tool timeout, trace persistence, concurrency, malformed
request, oversized request, and unknown route alerting in the release evidence.
The `observability` section also includes a ServiceMonitor semantic check for
Prometheus Operator clusters. It records `scrape_target_present`,
`selector_present`, missing scrape markers, and ServiceMonitor `sha256`, so the
release evidence proves the packaged monitor still selects
`self-correcting-agent` and scrapes `/metrics.prom` on the `http` port with the
expected interval and timeout.
The audit also emits `deployment` evidence with a Kubernetes manifest semantic
check. The Kubernetes manifest semantic check records
`required_resources_present`, `hardening_present`, `rollout_controls_present`,
resource count, missing resource kinds, missing hardening markers, missing
rollout markers, and manifest `sha256`, so release evidence proves the packaged
cluster manifest still contains the Secret, ConfigMap, PVC, Deployment, Service,
PodDisruptionBudget, NetworkPolicy, CronJob, production doctor, probes,
non-root/read-only security context, resource limits, graceful termination, and
trace retention job expected by internal production rollout.
The same `deployment` evidence includes a systemd unit semantic check for
single-host or VM rollout. It records `service_controls_present`,
`sandboxing_present`, `resource_controls_present`,
`trace_state_boundary_present`, missing marker lists, and unit file `sha256`,
so releases preserve the production doctor preflight, restart policy, process
sandbox, cgroup bounds, graceful shutdown window, and owner-only trace state
write boundary outside Kubernetes too.
GitHub Actions runs the same gate on Python 3.9 and 3.12 with read-only
repository permissions, then uploads the built wheel artifact for inspection.
CI uploads wheel and release manifest artifacts with 14-day artifact retention
so release evidence remains available long enough for review without relying on
repository-wide defaults.
The production doctor policy fails when required exposed-service controls are
missing: bearer auth, diagnostic endpoint protection, trace persistence,
per-client rate limiting, bounded run concurrency, or disabled full trace HTTP
responses. Placeholder bearer tokens fail with `auth_token_placeholder`, and
tokens that cannot be sent as safe HTTP header values fail with
`auth_token_unsafe`. `--require-auth` rejects placeholder tokens before a
weaker release gate can pass with example credentials. If full trace HTTP
responses are enabled, production doctor reports
`full_trace_response_must_be_disabled`.
Provider-backed deployments should add `--require-runtime-provider` to the
production doctor gate. That gate fails when `SELF_CORRECTING_LLM_BASE_URL`,
`SELF_CORRECTING_LLM_MODEL`, or `SELF_CORRECTING_LLM_API_KEY` is missing,
reporting `llm_base_url_required`, `llm_model_required`, or
`llm_api_key_required`. It also requires at least two runtime planning
iterations and reports `runtime_iterations_too_low` when the configured budget
cannot support one corrective replan. After the static provider gate passes,
run `scripts/smoke_real_llm_runtime.sh` as live evidence that CLI runtime,
HTTP `/runtime/run`, traces, metrics, policy-gated `http_request`, and
approval/resume work against the configured model. Provider-backed production
must attach that redacted JSON output to `scripts/production_readiness_audit.py`
with `--provider-smoke-evidence` and enforce it with
`--require-provider-smoke`; the audit records evidence hash and run IDs while
provider secrets stay in environment variables only.
The provider smoke evidence uses `evidence_schema_version: "1"` and must
include `approval_run_id`, `cli_run_id`, `http_run_id`, and `resumed_run_id`.
It also includes `provider_snapshot` fields such as `llm_base_url_host`,
`llm_model`, `llm_provider`, and `llm_api_key_configured`, plus
`capability_checks` for `cli_runtime`, `http_runtime`, `trace_status`,
`timeline`, `approval_resume`, and `metrics`. It also records
`runtime_effective_tool_policy_sha256`, binding the live model smoke to the
same runtime tool boundary reviewed by doctor, staging, and rollout sign-off
evidence. Missing required fields convert an otherwise passing provider smoke
file to `invalid_evidence`, blocking strict readiness and release-evidence
gates. It deliberately does not write the raw API key or full provider base URL.
Attach staging acceptance evidence with `--staging-acceptance-evidence` and
enforce it with `--require-staging-acceptance` before internal production
promotion. Pass the same `--require-provider-smoke` and
`--require-staging-acceptance` flags to
`self-correcting-agent-release-evidence` so the final approval bundle is also
blocked when external evidence is missing or incomplete. A passing staging file
must use `evidence_schema_version: "1"` and prove live `health_status`,
`ready_status`, `auth_subject`, `runtime_policy_source`, `runtime_run_status`,
`runtime_effective_tool_policy_count`,
`runtime_effective_tool_policy_sha256`, `runtime_note_allowed`,
`runtime_http_request_approval_required`, `runtime_run_id`,
`runtime_timeline_event_count`, `runtime_summary_run_count`,
`metrics_trace_persistence`, and `metrics_runtime_runs_total`; otherwise strict
gates mark it `invalid_evidence`.
Run `scripts/observability_acceptance.sh` against the deployed service with
`SELF_CORRECTING_OBSERVABILITY_BASE_URL` and a short-lived
`SELF_CORRECTING_OBSERVABILITY_TOKEN` when diagnostics are protected. Add
`SELF_CORRECTING_PROMETHEUS_BASE_URL` and, if needed,
`SELF_CORRECTING_PROMETHEUS_TOKEN` to prove Prometheus can query
`SELF_CORRECTING_PROMETHEUS_QUERY` with at least one result. Attach that
redacted JSON with `--observability-acceptance-evidence` and enforce it with
`--require-observability-acceptance` so promotion proves live `/metrics.prom`
exposes the required Prometheus metrics, the scrape/query path works, and the
packaged Grafana/Prometheus artifacts are still coherent. A passing
observability file must use `evidence_schema_version: "1"` and include
`metrics_status`, `required_metrics_present`, `required_metric_count`,
`metrics_sha256`, `grafana_dashboard_status`, `grafana_dashboard_sha256`,
`prometheus_rules_status`, `prometheus_rules_sha256`, and
`prometheus_query_status`; missing or invalid fields become `invalid_evidence`.
Before opening the service to internal teams, complete
`docs/internal-rollout.md` and attach internal rollout sign-off evidence for
team subjects, runtime tool policy, provider smoke, staging acceptance,
Prometheus/Grafana wiring, and rollback rehearsal.
Validate that sign-off with `scripts/internal_rollout_acceptance.py`; attach
its redacted output with `--internal-rollout-evidence` and enforce it with
`--require-internal-rollout`. The sign-off must include `tech_lead`, `sre`,
`security`, and `business_owner` roles plus checks for provider smoke, staging
acceptance, observability acceptance, runtime tool policy, team access, trace
retention, and rollback rehearsal. It must also bind approval to the reviewed
runtime tool boundary with `runtime_effective_tool_policy_sha256`.
The sign-off validator also records `expected_release_version`,
`version_matches`, `expected_environment`, and `environment_matches`; production
approval should reject stale sign-offs that target another version or
environment. A passing internal rollout evidence file must also include
`evidence_schema_version: "1"`, `rollout_id`, `signed_off_at_utc`,
`required_roles_present`,
`required_checks_passed`, `approver_role_count`,
`runtime_effective_tool_policy_sha256`, and the sign-off `sha256`;
strict gates downgrade incomplete sign-off evidence to `invalid_evidence`.
After all four external evidence files exist, run
`scripts/production_approval_bundle.sh --strict` or `make production-approval-bundle` to
generate the strict readiness audit and strict release evidence bundle with all
required evidence flags applied consistently. Override
`SELF_CORRECTING_PROVIDER_SMOKE_EVIDENCE`,
`SELF_CORRECTING_STAGING_ACCEPTANCE_EVIDENCE`,
`SELF_CORRECTING_OBSERVABILITY_ACCEPTANCE_EVIDENCE`,
`SELF_CORRECTING_INTERNAL_ROLLOUT_EVIDENCE`,
`SELF_CORRECTING_RELEASE_MANIFEST`, `SELF_CORRECTING_RUN_CHECKS_EXIT_CODE`,
`SELF_CORRECTING_READINESS_AUDIT_OUTPUT`, or
`SELF_CORRECTING_RELEASE_EVIDENCE_OUTPUT` when release automation stores
artifacts outside `/tmp`.
The bundle script reports all missing external evidence files as
`evidence_missing` with their labels and paths, then rejects evidence files older
than 24 hours by default before building the approval bundle. Set
`SELF_CORRECTING_EVIDENCE_MAX_AGE_SECONDS` only when the release process has an
explicit longer or shorter freshness window.
If that freshness window is not a positive integer, the bundle reports
`evidence_max_age_invalid` as structured JSON.
It also reports `release_manifest_missing` when the release manifest produced by
the standard gate is absent, before running readiness or release-evidence
checks.
Its stdout includes an `evidence_files` map with each external evidence path,
file name, byte size, `sha256`, `age_seconds`, `max_age_seconds`, and `fresh`
status so approvers can verify the bundle without opening raw evidence files.
The script accepts only the explicit `--strict` argument; unknown arguments fail
as structured `unknown_argument` JSON before evidence files are inspected.
When evidence files are present but fail strict semantic validation, the script
still writes readiness and release-evidence artifacts, prints redacted stdout
JSON with `status: "blocked"` and `failed_checks`, and exits with exit code 1.
Preflight errors such as missing evidence, stale evidence, invalid freshness
configuration, or a missing manifest are reported as stderr JSON before those
artifacts are built.
Run `scripts/staging_acceptance.sh` against the deployed staging URL with a
short-lived subject token to prove authenticated diagnostics, runtime policy,
deterministic runtime execution, persisted trace reads, fleet summaries, and
metrics work outside the local process.
Run `scripts/observability_acceptance.sh` after Prometheus/Grafana wiring to
prove the deployed `/metrics.prom` endpoint and packaged dashboard/rules are
ready for SRE review.

## Runtime Controls

- Optional bearer auth for `POST /run`.
- Optional named internal bearer tokens through
  `SELF_CORRECTING_SERVICE_AUTH_TOKENS`, with `auth_subject` access-log audit
  fields, per-subject rate-limit isolation, subject-scoped runtime trace reads,
  subject-scoped runtime resume, per-subject usage metrics, per-subject runtime trace audit,
  `resumed_by_auth_subject` approval audit, per-subject runtime resume metrics,
  per-subject runtime resume alerting, stale pending approval gauges, and
  per-subject runtime outcome metrics.
- Optional SQLite-backed idempotency cache via
  `SELF_CORRECTING_SERVICE_IDEMPOTENCY_CACHE_PATH` for restart-safe and
  same-volume replica retry reuse; leave unset for the in-memory per-process
  cache. `/ready` and `self-correcting-agent-doctor` validate
  `idempotency_cache_persistence` so bad cache volumes fail before traffic.
- Configurable runtime tool execution policy via
  `SELF_CORRECTING_SERVICE_RUNTIME_ALLOWED_TOOLS`; leave unset for the default
  policy, or set a comma-separated allowlist for tools that may execute without
  approval. Unknown tool names fail configuration before startup, and
  `/config`, `/metrics`, and Prometheus build info expose
  `runtime_allowed_tools` as non-secret audit metadata.
- Subject-level runtime tool execution policy via
  `SELF_CORRECTING_SERVICE_RUNTIME_ALLOWED_TOOLS_BY_SUBJECT` for internal
  teams that share one service but need different direct-execution boundaries.
  Matching `auth_subject` entries override the global allowlist, and audit
  surfaces expose `runtime_allowed_tools_by_subject_count` without exposing
  bearer tokens.
- Optional diagnostic endpoint bearer protection, required by production doctor
  gates while leaving health/readiness/version probes public.
- Request body size limit before agent execution.
- Per-client `/run` rate limit.
- `Retry-After` headers on rate-limit and run-concurrency rejections.
- `X-Forwarded-For` is ignored by default for rate limiting and must be
  explicitly trusted for reverse-proxy deployments.
- In-flight `/run` concurrency cap with JSON `503` responses.
- Bounded `/run` execution timeout with JSON `504` responses.
- Configurable `/run` goal length cap with JSON `413` responses.
- full trace responses are disabled by default with JSON `403` responses unless
  an operator explicitly enables the HTTP response surface.
- Configurable HTTP request read timeout for slow-client protection.
- Structured JSON access logs with request IDs, raw-key-free idempotency
  presence, and a documented access log schema for log pipeline ingestion.
- `X-Run-ID` and trace-aware `X-Trace-Path` response headers for client-side
  request, run, and persisted trace artifact correlation.
- `Cache-Control`, `X-Content-Type-Options`, `Referrer-Policy`,
  `Content-Security-Policy`, and `X-Frame-Options` response headers for cache,
  content-sniffing, referrer-leakage, browser execution, and framing hardening.
- `/config`, `/metrics`, and Prometheus `self_correcting_agent_build_info`
  expose `security_response_headers` and active security header policy values
  such as `content_security_policy_header` and `x_frame_options_header` for
  runtime rollout and gateway audits.
- `/config`, `/metrics`, and Prometheus `self_correcting_agent_build_info`
  expose trace permission policy fields for trace directories, trace files,
  and readiness probe files.
- `/config`, `/metrics`, and Prometheus `self_correcting_agent_build_info`
  expose redacted LLM provider audit fields, including `llm_provider`,
  `llm_base_url`, `llm_model`, `llm_api_key_configured`, and
  `llm_timeout_seconds`, plus retry policy fields `llm_max_retries` and
  `llm_retry_backoff_seconds`, without exposing the raw provider key.
- OpenAI-compatible provider calls can retry transient 429 and 5xx failures
  through `SELF_CORRECTING_LLM_MAX_RETRIES` and
  `SELF_CORRECTING_LLM_RETRY_BACKOFF_SECONDS`; defaults use a small retry
  budget, and numeric provider `Retry-After` response headers take precedence.
- Production HTTP server defaults: bounded request threads with
  `block_on_close`, address reuse, and explicit listen backlog for graceful
  shutdown, restart, and burst tolerance.
- `SIGTERM` handling for orchestrator shutdown with HTTP server close and exit
  status `143`.
- Kubernetes termination grace period longer than the default run timeout.
- `HEAD /health` and `OPTIONS /run` support for load balancers, probes, and
  client method discovery.
- JSON `405` responses include `Allow: GET, HEAD, OPTIONS, POST`.
- `/metrics` runtime counters by path/status code, error code, average/max
  request duration, agent run outcomes/duration, uptime, active/max concurrent
  run gauges, active rate-limit window gauges, authenticated internal subject
  usage counters, idempotency cache activity and eviction gauges, bind host and
  bind port audit fields, max request bytes gauges, trusted-forwarded-for
  rollout state, and redacted runtime build info.
- `/metrics.prom` Prometheus-compatible text exposition for scrape-based
  monitoring, including a request duration histogram, an agent run duration histogram,
  a runtime run duration histogram, and `self_correcting_agent_build_info` for
  rollout audits.
- runtime tool timeout enforcement with planner-visible `timeout_seconds` and
  structured `tool_execution_timeout` observations for bounded replanning.
- Policy-gated `http_request` applies SSRF defense in depth by rejecting
  private, loopback, and link-local targets before opening a socket.
- Optional full-trace artifact persistence with atomic file replacement.
- `/ready` validates configured trace persistence with an owner-only
  temporary-file write/delete probe.
- `/ready` validates configured SQLite idempotency persistence before reporting
  the service ready.
- `/ready` reports structured `failed_checks` for dependency-level triage
  without requiring operators to parse check messages.

## Deployment Artifacts

- `Dockerfile` for container runtime.
- `.dockerignore` to keep local and generated artifacts out of images.
- `deploy/env.example` for service environment variables.
- `docs/deployment.md` for Docker and runtime operation.

  per-subject runtime resume alerting, stale pending approval alerting,

  rate-limit, concurrency saturation,
  idempotency conflict, request body timeout, malformed run request,
  oversized run request, suspicious HTTP framing, and unknown route alerting.
- `deploy/prometheus/self-correcting-agent-servicemonitor.yaml` for
  Prometheus Operator clusters that scrape targets through `ServiceMonitor`
  resources.

## Known External Dependencies

- Docker image build requires a running Docker daemon.
- Kubernetes deployment requires a cluster, an immutable image tag, a real
  bearer token Secret, and storage that can satisfy the `ReadWriteMany` trace
  PVC used by the two-replica manifest.
- Public exposure should sit behind a reverse proxy for TLS, network policy,
  connection limits, and request timeouts.
- Provider-backed runtime planning requires an OpenAI-compatible chat
  completions endpoint. Use `--require-runtime-provider` plus
  `scripts/smoke_real_llm_runtime.sh` before promoting a deployment that relies
  on real LLM planning.
